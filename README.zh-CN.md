# IntentGraph

[English](./README.md)

IntentGraph 是一个面向游戏逻辑的最小 Blueprint/Graph 框架。

这个项目会刻意从小处开始。近期目标不是直接做成一个庞大的可视化脚本平台，而是先建立一个可以持续扩展的核心闭环，并在真实游戏与 UGC 需求中逐步演进。

## AI 友好的 Graph

IntentGraph 不是想做一个传统意义上的蓝图系统。

它更准确的定位，是一个面向游戏逻辑、对 AI 更友好的 graph：它是结构化的、文本优先的、schema 驱动的，便于 AI 读取、生成、patch、校验和扩展，而不是把能力中心放在一个纯编辑器驱动的工作流里。

在当前这个 AI 发展阶段，这个方向的目标是让游戏制作更高效、更实际。我们希望借助 AI 更好地协作生产游戏逻辑，而不是先去复刻一整套旧形态的可视化脚本工具。

## 项目愿景

IntentGraph 聚焦于一种面向游戏逻辑的、以文本为优先的表达方式：

- 用文本描述 graph
- 定义节点 schema
- 通过 registry 注册节点
- 对 graph 结构和数据做校验
- 用最小执行器驱动运行时逻辑
- 通过 patch 更新 graph

第一阶段的主要场景是机关触发类逻辑，例如装置、机关、关卡交互事件等。

## 当前目标

当前阶段的目标是做一个最小的 `blueprint-core`，包括：

- 文本化 graph 描述
- 节点 schema 定义与注册
- 基础校验
- 最小执行器
- example 与 patch 支持
- 对 AI 辅助新增节点友好的工作流

## 产品方向

IntentGraph 不会一开始就去设计一个大而全的节点系统，而是会在真实 UGC 需求中逐步补充节点。

这意味着：

- 核心保持足够小且可扩展
- 新节点应当由真实玩法需求驱动产生
- 随着节点数量增长，数据格式、校验和执行模型仍然保持稳定

## 当前不做

IntentGraph 目前明确不追求以下方向：

- 传统蓝图系统的复刻版
- GUI 编辑器
- 完整 UGC 平台
- 通用脚本语言
- 大而全的节点库
- 复杂子图或继承系统
- 对标 Unreal Engine Blueprint 的完整能力

## 为什么这样设计

很多游戏逻辑系统的问题，不是能力不够，而是过早做得太大、太泛。这个项目希望反过来：

- 先把最小闭环做通
- 先在一个具体玩法类型里验证
- 只在真实使用暴露需求后再继续扩展

这背后还有一个判断：在当前阶段，最有用的游戏逻辑 graph 不一定是最“可视化”的那个，而是最容易让人和 AI 一起编写、检查、patch、校验和执行的那个。

相比一个边界模糊但规模庞大的框架，一个小而可靠的核心往往更有价值。

## 第一阶段聚焦

第一阶段主要围绕触发器和机关类逻辑，例如：

- trigger -> condition -> action
- 装置激活链路
- 关卡交互事件
- 简单事件驱动玩法流程

这个范围是有意收窄的。它足以先验证 `schema / graph / registry / validator / executor / patch` 这条核心链路，再决定后续是否进入更复杂的系统。

## 当前状态

当前仓库重点是先明确框架基础能力与演进方向。优先级是先做出可扩展的核心闭环，再逐步补充节点集合和外围工具。
